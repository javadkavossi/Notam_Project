## 🎯 چشم‌انداز فاز ۱

خلبان/اپراتور در پنل خود یک **پرواز** تعریف می‌کند:

```
مبدأ (ADEP) + مقصد (ADES) + الترنت‌ها (ALTN) + روت (FIRها/waypointها) + پنجرهٔ زمانی پرواز
        └──────────────── حدوداً ۵ فرودگاه + مسیر ────────────────┘
```

سیستم باید **تمام NOTAMهای مرتبط** با این پرواز را:
1. **پیدا کند** — مکانی (فرودگاه‌ها + FIRهای مسیر + تقاطع جغرافیایی روت) و زمانی (هم‌پوشانی اعتبار NOTAM با پنجرهٔ پرواز).
2. **تحلیل و دسته‌بندی کند** — این NOTAM به چه بخشی مربوط است (باند/تاکسی‌وی/ناوبری/فضای هوایی/…) و چه اطلاعاتی دارد.
3. **امتیاز اهمیت بدهد** — درصد/سطح اهمیت هر NOTAM تا خلبان سریع‌تر مورد بحرانی را ببیند.
4. **برجسته و مرتب نمایش دهد** — موارد بحرانی بولد و بالای لیست؛ رابط سریع و جذاب.
5. **زنده هشدار دهد** — اگر در پنجرهٔ پرواز NOTAM جدیدی آمد، فوراً اطلاع دهد؛ اگر بحرانی بود، اخطار اجباری.

**فازهای بعدی (خارج از دامنهٔ این مستندات، فقط برای جهت‌گیری معماری):** اتصال به LLM برای گزارش بهتر، بریفینگ صوتی مختصر، تقویت بصری/شماتیک.

## 🧭 اصول طراحی (Design Principles)

این سیستم safety-critical است. هر تصمیم معماری با این اصول سنجیده می‌شود:

1. **هیچ NOTAMی نباید گم شود (No Silent Loss).** دریافت تضمین‌شده، ack پس از ذخیره، صف داخلی پایدار، تشخیص gap و backfill. اگر داده‌ای از دست رفت، سیستم باید **بداند و اعلام کند** — سکوت ممنوع.
2. **اگر سیستم نمی‌داند، باید بگوید نمی‌داند.** قطع ارتباط با هر منبع، عقب‌افتادگی پردازش، یا کهنگی داده باید در UI و مانیتورینگ **صریح** باشد. بریفینگ هرگز نباید «کامل» به نظر برسد وقتی منبعی down است.
3. **دقت بر سرعتِ ناقص مقدم است، اما هر دو لازم‌اند.** دیکد Q-code و دسته‌بندی قاعده‌محور (deterministic) پایهٔ دقت است؛ کش و ایندکس‌گذاری پایهٔ سرعت.
4. **اجماع چندمنبعی (Consensus).** هر NOTAM یک هویت متعارف (canonical) دارد؛ هر منبع یک «مشاهده» ثبت می‌کند. تعدد منابع = اطمینان بیشتر؛ تعارض = پرچم برای بازبینی.
5. **تکاملی، نه انقلابی.** روی همین کد Go/React ساخته می‌شود؛ ماژولار می‌شود تا بعداً قابل تفکیک به سرویس باشد. بدون بازنویسی از صفر.
6. **قابل‌مشاهده بودن (Observability) از روز اول.** متریک، health، audit trail و alert جزو الزامات پایه‌اند، نه افزودنی بعدی.
7. **داده مرجع، منبع حقیقت واحد و نسخه‌دار.** فرودگاه/باند/ناوید/مرز FIR باید دقیق، کامل و نسخه‌بندی‌شده باشند و تغییراتشان ردیابی شود.

## 🏛️ تصمیمات کلیدی (خلاصه)

بر اساس جمع‌بندی با کارفرما:

| موضوع | تصمیم | دلیل |
|-------|-------|------|
| سبک معماری | **مونولیت ماژولار** (Go) | تکاملی روی کد موجود، ریسک کم، تیم کوچک، قابل تفکیک بعدی |
| منابع NOTAM | FAA SWIM (زنده) + **AFTN** (کاندید بعدی) + FAA REST API (reconciliation) | چندمنبعی برای اطمینان و عدم‌ازدست‌رفتن؛ فعلاً FAA زنده |
| مقیاس | متوسط (صدها کاربر) | تعیین‌کنندهٔ انتخاب صف و کش |
| صف داخلی | **Redis Streams** (consumer group + replay) | on-prem، موجود در استک، تضمین at-least-once؛ مسیر ارتقا: NATS JetStream |
| داده مکانی | **PostgreSQL + PostGIS** | تطبیق روت/FIR جغرافیایی دقیق و سریع |
| بلادرنگ به فرانت | **WebSocket** (جایگزین polling) | تأخیر کم، فشار کمتر روی سرور |
| استقرار | **on-prem** | بدون وابستگی به managed cloud |

جزئیات و بده‌بستان‌ها در [ARCHITECTURE.md](ARCHITECTURE.md).

## 📖 واژه‌نامه (Glossary)

| اصطلاح | توضیح |
|--------|--------|
| **NOTAM** | Notice to Airmen — اطلاعیهٔ رسمی دربارهٔ وضعیت تأسیسات/خدمات/فضای هوایی |
| **ADEP / ADES** | Aerodrome of Departure / Destination — فرودگاه مبدأ/مقصد |
| **ALTN** | Alternate — فرودگاه جایگزین (اضطراری) |
| **FIR** | Flight Information Region — منطقهٔ اطلاعات پروازی (بلوک فضای هوایی) |
| **Q-line / Q-code** | خط Q در NOTAM حاوی کد ۵حرفی که موضوع و وضعیت را ماشین‌خوان می‌کند (مثل `QMRLC`) |
| **PIB** | Pre-flight Information Bulletin — بولتن اطلاعات پیش‌پروازی (خروجی بریفینگ) |
| **FICON** | Field Condition — گزارش وضعیت سطح باند (آلودگی/اصطکاک)؛ بسیار مهم برای فرود |
| **NOTAMN / NOTAMR / NOTAMC** | New / Replace / Cancel — نوع رویداد NOTAM |
| **SWIM** | System Wide Information Management — بستر توزیع دادهٔ FAA |
| **AFTN** | Aeronautical Fixed Telecommunication Network — شبکهٔ مخابراتی ثابت هوانوردی (منبع کاندید) |
| **Gap detection** | تشخیص شمارهٔ سریال‌های جاافتاده در یک سری NOTAM برای کشف داده‌های ازدست‌رفته |
| **Reconciliation** | تطبیق دوره‌ای دادهٔ ذخیره‌شده با pull کامل از منبع مرجع برای backfill |
| **Briefing Engine** | موتور تطبیق NOTAMهای مرتبط با یک پرواز + دسته‌بندی + رتبه‌بندی |
| **Provenance / Sighting** | ثبت اینکه هر NOTAM از کدام منبع و چه زمانی دیده شده (برای اجماع) |
| **DLQ** | Dead-Letter Queue — صف پیام‌هایی که پردازش نشدند (برای بازبینی، نه دورریختن) |

## ✅ معیار «تمام‌شدن» فاز ۱ (Definition of Done)

فاز ۱ وقتی کامل است که:

- [ ] کاربر بتواند پرواز (۵ فرودگاه + روت + زمان) را در پنل تعریف کند و بریفینگ کامل و مرتب‌شده بگیرد.
- [ ] هر NOTAM به‌صورت قاعده‌محور دسته‌بندی و امتیازدهی شود و موارد بحرانی برجسته شوند.
- [ ] هیچ NOTAMی به‌صورت خاموش گم نشود؛ gap detection و reconciliation فعال باشد.
- [ ] وضعیت سلامت هر منبع و lag پردازش در UI و مانیتورینگ صریح باشد.
- [ ] NOTAM جدیدِ مرتبط با پرواز فعال، زنده (WebSocket) و در صورت بحرانی با اخطار، اطلاع داده شود.
- [ ] دادهٔ مرجع (فرودگاه/باند/ناوید/FIR) کامل، نسخه‌دار و با تشخیص تغییر باشد.
- [ ] معماری چندمنبعی آماده باشد تا افزودن AFTN فقط «یک آداپتور جدید» باشد، نه تغییر هسته.
